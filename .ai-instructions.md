# Formative-Assessment Based Learning Management System - AI Assistant Core Context

## 1. Project Identity & Core Philosophy
FabLMS is an Open Source (GPLv2) platform designed to run school information 
systems with a strict pedagogical focus on **Formative Assessment** and 
**Demonstration of Learning**.

### Core Architectural Directives based on Philosophy:
*   **No Summative Endpoints:** Do not design standard "Pass/Fail" or simple integer-grade systems. Assessments must always link to specific, trackable "Skills".
*   **Skill-Based Grading:** A grade is a collection of assessed skills with individual weights, tied to a `ClassCriteria`.
*   **Soft Deletes:** Never permanently delete users or critical records; always use Laravel's SoftDeletes to maintain historical assessment links.

### Strict Domain Vocabulary (Do not use standard LMS terms):
When generating models, controllers, or UI text, strictly adhere to these terms:
*   **NEVER USE:** "Homework", "Assignment", "Test", "Quiz".
*   **USE `LearningDemonstration`:** The overarching model/concept describing what a student must do.
*   **USE `LearningDemonstrationTemplate`:** A reusable, shareable blueprint of a demonstration, stripped of class/student links.
*   **USE `LearningDemonstrationOpportunity`:** The actual instance of a demonstration assigned to a specific student/class for them to complete.
*   **USE `AssessmentStrategy`:** The logic defining how a specific `ClassSession` calculates its grades based on `ClassCriteria`.
*   **USE `DemonstrationTypes`:** A polymorphic class (NOT an enum) to allow integration of future types (Standard, Recurring, Peer Review).

## 3. Architectural Guidelines & Laravel Conventions

### 1. Custom Classes (`app/Classes`)
- **Structure:** Classes are organized into feature-based subdirectories (e.g., `Integrators`, `Storage`, `Settings`, `People`).
- **Standard Patterns:**
    - Use **PHP 8 Constructor Property Promotion** for dependencies.
    - Implement the `App\Interfaces\Synthesizable` interface for objects that need to be converted to/from JSON (e.g., `NameConstructor`).
    - Frequently use a static `instance()` method for singleton-like access via the Laravel Service Container (e.g., `SessionSettings::instance()`).
- **Invocation:** Typically invoked via dependency injection in Livewire/Controllers or through the service container when global state/configuration is involved.

### 2. Polymorphic Integrations (`app/Models/Integrations`)
- **Polymorphic Base Classes:** `Integrator`, `IntegrationService`, and `IntegrationConnection` act as base models.
- **Subclass Instantiation:** They override the `newFromBuilder` method to automatically return a specialized subclass based on the `className` attribute stored in the database.
    - *Example:* `Integrator::all()` returns a collection of `LmsIntegrator` subclasses.
- **Abstract Implementation:** Base classes (prefixed with `Lms`) are `abstract` and define mandatory static methods for metadata (`getPath()`, `integratorName()`, `defaultData()`).
- **Connections:** Specialized connection models are stored in `app/Models/Integrations/Connections/` (e.g., `AiConnection`, `SmsConnection`).

## 3. Architectural Guidelines & Laravel 12 Conventions

*   **Controllers:**
    *   **Skinny controllers only:** Limit strictly to standard CRUD methods (`index`, `show`, `create`, `store`, `edit`, `update`, `destroy`).
    *   **Inline Validation:** Perform validation directly inside `store/update` methods using `$request->validate([...], static::errors())`.
    *   **Authorization:** Consistently use `Gate::authorize('action', $model)` at the beginning of methods.
    *   **Middleware:** Define middleware via the modern static `middleware()` method within the controller class.
    *   **State:** Use `App\Classes\SessionSettings` to manage and persist UI state across requests.

*   **Database & Eloquent Patterns:**
    *   **No Repositories:** Keep logic in Models (via scopes/relationships) or extract to `app/Classes`.
    *   **Scopes:** Extensively use Eloquent scopes for filtering. In Laravel 12, use the `#[Scope]` attribute on protected methods (e.g., `#[Scope] protected function enabled(Builder $query)`).
    *   **Casts:** Always define model attribute casts using the modern `casts(): array` method rather than the legacy `$casts` property.
    *   **Relationships:** Always use explicit return type hints (e.g., `public function services(): HasMany`).
    *   **Soft Deletes:** Always implement the `SoftDeletes` trait for critical records to maintain historical assessment links.

*   **Custom Classes (`app/Classes`):**
    *   Organize classes into feature-based subdirectories (e.g., `Integrators`, `Storage`, `Settings`, `People`).
    *   Always use PHP Constructor Property Promotion for dependencies.
    *   Implement the `App\Interfaces\Synthesizable` interface for objects needing JSON conversion.
    *   Use a static `instance()` method for singleton-like access via the Service Container when necessary.

*   **Integrations Architecture (Polymorphic Subclassing):**
    *   The `Integrator`, `IntegrationService`, and `IntegrationConnection` models act as base models.
    *   They MUST override the `newFromBuilder` method to automatically return a specialized subclass based on the `className` attribute stored in the DB.
    *   Base classes (prefixed with `Lms`) are `abstract` and must define mandatory static methods for metadata (`getPath()`, `integratorName()`, `defaultData()`).

*   **Internationalization (i18n):**
    *   Never hardcode strings. Use `__()` and `trans_choice()` exclusively.
    *   Use **dot notation** for keys (e.g., `__('auth.login_error')`).
    *   Store translation arrays in `lang/{locale}/{namespace}.php`.

## 4. UI/UX, Frontend, & Livewire Standards

*   **Styling & Structure:**
    *   Strictly use **Bootstrap 5** utility classes and standard form controls. Do not write custom CSS unless absolutely required.
    *   Use **Alpine.js** for lightweight, client-side DOM manipulation (modals, toggles).
    *   Use **Livewire 4** for all stateful UI, complex data tables, and backend interactions.
    *   Avoid putting PHP code in views. Move that PHP code to the controller, or the code section of the Livewire component.

*   **Livewire Component Conventions:**
    *   **Validation:** Validation logic must be exclusively defined within the `rules()` method of the component. NEVER use Form Requests.
    *   **Derived State:** Always use the `#[Computed]` attribute (via `Livewire\Attributes\Computed`) for derived state properties.
    *   **Hydration:** Utilize custom `Synth` classes (stored in `app/Classes/Synths`) to handle complex object serialization/hydration in the frontend.

## 5. Strict Anti-Patterns (What NOT to do)

1.  **Vocabulary Violations:** NEVER use the terms "Homework", "Assignment", "Test", or "Quiz". Strictly use `LearningDemonstration`, `DemonstrationOpportunity`, etc.
2.  **FormRequests:** NEVER generate or use Laravel Form Requests (`App\Http\Requests\...`).
3.  **Directory Violations:** NEVER create or use `app/Services` or `app/Actions`. All custom business logic belongs in `app/Classes`.
4.  **Legacy Queries:** NEVER use raw SQL queries if Eloquent relationships or `#[Scope]` attributes can achieve the result.
5.  **Legacy Frontend:** AVOID jQuery entirely unless writing a specific wrapper for a mandatory legacy plugin.ple: "Do not suggest adding heavy frontend JS frameworks like React/Vue unless asked. Do not write raw SQL queries if Eloquent/Query Builder can handle it efficiently. Do not ignore Laravel's built-in validation mechanisms.")*
